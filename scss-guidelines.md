# SCSS構成・運用ルール  v0.0.0

本テーマの CSS は、**保守性・再利用性・属人化防止**を目的として設計されています。
非エンジニアや将来の担当者、AI アシスタントが迷わず安全に運用できるよう、以下のルールに従ってください。

---

## 要点まとめ（まずここだけ読めばOK）

このドキュメントは、**保守性・再利用性・属人化防止**を目的とした SCSS 設計・運用方針の提案書です。

### 設計の前提（4つ）

- **SCSS で管理**：サイトの見た目（CSS）は、必ず `scss/` フォルダ内のファイルだけを編集します。最終的に 1 つの `style.css` にまとまります。
- **フォルダは役割で分ける**：土台（foundation）・枠組み（layout）・部品（object）の 3 階層で構成します。部品の名前付けには **BEM**（ブロック・要素・バリエーションを区切るルール）を使います。
- **部品はどこでも使えるようにする**：ボタンやカードなど部品（c-）は、特定のページやレイアウトに依存しないように書きます。そうすると、どこにでも配置しやすくなります。
- **p-（ページ用）はそのページ専用。流用したくなったら c-（部品）に昇格させる**。

### フォルダの役割（どこに何を書くか）

| フォルダ | 役割 | たとえば |
| :--- | :--- | :--- |
| **foundation** | 設計の土台。色・フォントの基準値、共通のリセットなど | 色の定義、文字サイズの計算ルール |
| **layout** | サイト全体の枠組み | ヘッダー、フッター、サイドバー |
| **object/component** | どこでも使い回せる部品 | ボタン、カード、見出し |
| **object/page** | そのページだけの装飾 | トップページ用、お問い合わせページ用 |
| **object/utility** | どうしても 1 箇所だけ調整したいときの最終手段 | 余白の微調整、文字の中央寄せ |

### 数値・色のルール

- **大きさは「rem」で書く**：ピクセル（px）ではなく、rem という単位を基本にします。画面サイズに応じて適切に伸び縮みしやすくなります。
- **文字サイズ**：専用の関数 `fluid-font-rem()` を使います。
- **余白・幅**：スマホと PC で違う値にしたいときだけ `fluid-space-rem()` を使い、同じでよいときは `rem` で書いてよいです。
- **線の太さ（1px）**など、デザイン上 px の方が適切な場合は px を使ってよいです。
- **色の濃さは数で表す**：例として `--c-gray-500` のように数値を使います。「暗いグレー」「薄いグレー」のような主観的な名前は使わず、文字の太さ（font-weight）のように数値で段階を表す考え方に統一します。

### 表示の考え方（スマホ・PC）

- **まずスマホ用を書く**（モバイルファースト）。そのうえで、画面が広いとき用のスタイルを足していきます。
- デザインが PC 用で渡されても、CSS ではスマホ用を「基本」として整理してから、PC 用の差分だけを追加します。

### AI にスタイルを依頼するとき

- **フロント用（サイトの見た目）**：`style.scss` には直接書かず、必ず「部品用のファイル」を作り、一覧ファイル（`_index.scss`）に登録する形にします。
- **管理画面用**：`editor.scss` は、管理画面だけの見た目（ブロックエディタ用など）が必要な場合に限り、直接書いてよいです。
- AI に依頼するときは「このドキュメントの設計思想に従って」と伝えると、ルールに沿ったコードが出やすくなります。

## 1. CSS設計思想

### 基本の考え方

- **SCSS で管理する**  
  実際にブラウザで読み込まれるのは `style.css` ですが、**編集するのは必ず `scss/` フォルダ内のファイル**にします。SCSS は CSS を書きやすくするための仕組みで、ここで書いた内容が自動的に `style.css` に反映されます。

- **フォルダは役割で 3 階層に分ける（FLOCSS）**  
  「土台（foundation）」「枠組み（layout）」「部品（object）」の 3 つに分けます。そうすることで、「どのファイルに何が書いてあるか」が分かりやすくなります。

- **部品の名前は BEM で付ける**  
  部品（c- で始まるクラス）の中では、**BEM** という名前の付け方を使います。
  - **Block（ブロック）**：部品そのもの。例：`.c-card`（カード全体）
  - **Element（要素）**：部品の「中のパーツ」。例：`.c-card__title`（カードのタイトル）
  - **Modifier（修飾）**：「バリエーション・状態」だけに使う。例：`.c-card--featured`（目立たせるカード）
  
  「タイトル」「画像」のような**構造上のパーツ**は `__` で区切った Element にします。「強調」「無効」のような**見た目のバリエーション**だけ `--` で区切った Modifier にします。

- **ブロックをまたいでネストしない**  
  p-（ページ用）を親にして c-（部品）を縛る書き方は避けます（依存が増え、部品をどこでも使えなくなるため）。c- の**中**では、同じ部品の要素として `&__element` のネストで書いてよいです。

- **1 ファイル 1 つの役割**  
  1 つのファイルには、1 種類の役割のスタイルだけを書くようにします。ファイル名を見れば「何が書いてあるか」が分かるようにします。

### フォルダ構成

```text
scss/
├─ style.scss        … フロント用の「入口」ファイル（ここに直接スタイルは書かない。部品ファイルを作って登録する）
├─ editor.scss       … 管理画面用の「入口」ファイル（管理画面だけの見た目は、必要なときだけここに書いてよい）
├─ foundation/       … 土台（色・フォントの基準、リセットなど。l- / c- などの部品クラスは置かない）
├─ layout/           … 枠組み（ヘッダー・フッター・サイドバーなど）
└─ object/
   ├─ component/     … 部品（ボタン・カード・見出しなど）
   ├─ page/          … ページごとの装飾（トップページ、記事詳細など）
   └─ utility/       … 微調整（余白の調整、表示の切り替えなど。使うのは最終手段として）
```

## 2. 新しくスタイルを追加する手順

「このスタイルはどこに書けばよいか」迷ったときは、**下の表の優先順位**で置き場所を決めてください。

| 優先順位 | どんなときか | フォルダ | たとえば |
| :--- | :--- | :--- | :--- |
| **1** | 他のページでも使いそうな部品 | `object/component` | ボタン、カード、一覧の 1 行 |
| **2** | そのページだけのデザイン | `object/page` | トップページの特設バナー、お問い合わせページのレイアウト |
| **3** | サイト共通の枠 | `layout` | ヘッダー、フッター、共通ナビ |
| **4** | どうしても 1 箇所だけ直したい | `object/utility` | 「ここだけ余白を足す」「ここだけ非表示」など（最終手段） |

### 作業の流れ（3ステップ）

1. **新しいファイルを作る**  
   例：カードのスタイルなら `object/component/_card.scss` のようなファイルを新規作成します。

2. **一覧に登録する**  
   同じ階層にある `_index.scss` を開き、さきほど作ったファイルを「読み込む」1 行を追加します。**原則は `@forward` で登録**します。  
   例：`object/component/_index.scss` に `@forward "component/card";` と書く。  
   ※ 変数や mixin を参照する側のファイルでは `@use` を使います。新規の部品を登録するときは `@forward` で十分です。

3. **ここまでで完了**  
   `style.scss` をいじる必要はありません。登録したファイルは自動的にまとめられて反映されます。

## 3. 命名ルール（接頭辞）

クラス名の**先頭に、役割が分かる記号**を付けます。そうすると、見ただけで「レイアウト用か」「部品か」「ページ用か」が判断しやすくなります。

| 記号 | 意味 | 使う場面 | 例 |
| :--- | :--- | :--- | :--- |
| **l-** | Layout（レイアウト） | サイトの枠組み | `.l-header`（ヘッダー）, `.l-main`（メインエリア） |
| **c-** | Component（コンポーネント＝部品） | 使い回す部品 | `.c-btn`（ボタン）, `.c-card`（カード） |
| **p-** | Page（ページ） | そのページ専用 | `.p-home`（トップ用）, `.p-contact`（お問い合わせ用） |
| **u-** | Utility（ユーティリティ＝調整） | ちょっとした調整 | `.u-mt-10`（上に余白）, `.u-text-center`（文字中央） |

## 4. Figma → SCSS 変換ルール

Figma のデザインをコードに落とすときのルールです。

### 文字サイズ・余白

**基本は「rem」で書く**（px はなるべく使わない）。rem は画面の大きさに合わせて適切に伸び縮みしやすい単位です。

- **文字サイズ**：`fluid-font-rem()` という関数を使います。
- **余白・幅**：スマホと PC で**値が変わる**ときだけ `fluid-space-rem()` を使います。**同じ値でよい**ときは `rem` で書いてよいです。
- **線の太さ（1px）**など、デザイン上 px の方が適切な場合は px を使ってよいです。

Figma に書いてある数値（px）は、次の表を目安に変換してください。

| 種類 | Figma の例（px） | SCSS での書き方 | 補足 |
| :--- | :--- | :--- | :--- |
| 文字サイズ（ずっと同じ） | 16px | `1rem` | 16px ＝ 1rem として扱う |
| 文字サイズ（SP と PC で変える） | 24px | `fluid-font-rem(1.5, 2)` | 画面幅に応じて変わる |
| 余白・幅 | 24px | `padding: fluid-space-rem(1.5, 2);` など | 振り幅があるときはこの関数を使う |

### 色（カラー）

**色コード（#333 など）をそのまま書かず、色に「名前」を付けて使う**ルールにします。色の定義は `foundation/_color.scss` にまとめ、そこに書いた名前（トークン）を参照して使います。

- **濃さは数値で表す**  
  例：`--c-gray-500` のように数値で段階を付けます。「暗いグレー」「薄いグレー」のような主観的な名前（`--c-gray-dark` など）は使わず、文字の太さ（font-weight）のように数値で表す考え方に統一します。

- **役割が決まっている色は例外でよい**  
  「メイン色」「アクセント色」のように、デザイン上必ずある色（`--c-main`, `--c-accent` など）は、その名前を使ってよいです。

- **「意味」を持たせたい色**  
  「エラー用の赤」のように意味を持たせたいときは、まず「赤」のような色名トークンを定義し、その色を継承して「エラー用」のような名前を付けます。  
  例：`--c-error: var(--c-red);`（エラー色は、定義した赤を使う）

※ **色コード（#e53935 など）を書いてよいのは `foundation/_color.scss` の定義だけ**です。それ以外のファイルでは、色の名前（トークン）だけを参照して使います。

**悪い例・良い例**

```scss
/* 悪い例：色を数字で直接書いている */
color: #333;
background-color: #fff;

/* 良い例：色の名前（トークン）を使う */
color: var(--c-gray-900);
background-color: var(--c-white);

/* 良い例：「意味」を持たせる色は、色名を継承して定義する */
/* foundation/_color.scss の例 */
:root {
  --c-red: #e53935;
  --c-error: var(--c-red);   /* エラー用 ＝ 赤、という意味づけ */
}
```

## 5. レスポンシブのルール

**まずスマホ用を書き、そのうえで PC 用を足していく**（モバイルファースト）で実装します。

> **補足**  
> デザインが PC 用で渡されることが多いですが、CSS では「スマホ向けに整理した見た目」を**最初に**書き、そのうえで「画面が広いときはこうする」という差分だけを追加します。そうすると、スタイルの見通しが良くなり、修正もしやすくなります。

### 基本方針

- **最初に書くスタイル ＝ スマホ（SP）用**
- タブレット・PC 用は、`@include tb` のように「画面がこれ以上広いとき」の条件で**上書きや追加**を書きます。
- 「PC 用を先に書いて、スマホで打ち消す」書き方は避けます。上書きが増えて、分かりにくくなります。

**このやり方のメリット**

- どのスタイルがどの画面用か、追いやすい
- 無駄な上書きが減り、修正しやすくなる
- 新しい画面サイズにも対応しやすい

### 書くときのルール

1. スマホ用のスタイルを、ふつうの CSS として先に書く
2. 画面が広いとき用は、「〇px 以上のとき」のような条件（メディアクエリ）で**足すだけ**にする

### 記述例（イメージ）

```scss
.c-archive-list {
   display: flex;
   flex-direction: column;
   gap: fluid-space-rem(1, 1.5);
   li {
      border: 1px solid var(--c-blue);
   }

   /* タブレット以上の幅のとき */
   @include tb {
      flex-direction: row;
      flex-wrap: wrap;

      li {
         width: calc((100% - (4 * #{fluid-space-rem(1, 1.5)})) / 5);
      }
   }
}
```

（まず縦並び・スマホ向けを書いてから、`@include tb` のなかで「横並び・幅の計算」を追加しているイメージです。）

## 6. AIアシスタントへの指示ルール

AI（ChatGPT, Claude, GitHub Copilot など）にスタイルの修正を依頼するときは、**下の枠内の文章をそのままコピーして一緒に渡す**と、このプロジェクトのルールに沿ったコードが出やすくなります。

```markdown
# AIへの指示ルール
このプロジェクトのCSS設計ルールに従ってコーディングしてください。

## 重視する制約
1. **ディレクトリ構造**: FLOCSSベース（foundation / layout / object）を厳守すること。
2. **命名（FLOCSS × BEM）**: レイヤーは FLOCSS（foundation / layout / object）を守り、Component（c-）内は BEM（Block__Element--Modifier）で命名すること。`--Modifier` は状態/バリエーションにのみ使用すること。
3. **エントリポイント**: `style.scss` に直接スタイルを書かないこと。必ず部分ファイルを作成し `@use` または階層ごとの `_index.scss` で `@forward` すること。`editor.scss` は管理画面固有のスタイル（ブロックエディタ専用の見た目など）が必要な場合に限り、直書きしてよい。
4. **単位の扱い**: `px` 指定はなるべく避け、原則として rem ベースで記述すること。文字サイズは `fluid-font-rem()`。余白・レイアウト幅は SP/PC で値が変わる場合に `fluid-space-rem()` を使用し、固定の場合は `rem` でよい。境界線（1px）など、デザイン上 px が適切なケースは許容する。
5. **色（トークン）**: 色コードの直書きは避け、原則は色名トークン（例：`--c-red`, `--c-gray-500`）を使うこと。濃さ（明るさ・暗さ）は数値で表し、`--c-gray-dark` / `--c-gray-light` のような主観的な名前は使わないこと（font-weight のように数値化する考え方に統一）。`--c-main` / `--c-accent` など必ず存在する役割トークンは例外として可。意味トークン（例：`--c-error`）は色名トークンを継承して定義すること（例：`--c-error: var(--c-red);`）。
6. **命名規則**: クラス名には適切なプレフィックス（l-, c-, p-, u-）を付与すること。
7. **依存関係**: Component（c-）が Layout（l-）や Page（p-）に依存するスタイルを書かないこと。Component はどこでも配置可能にすること。
8. **関数定義**: `fluid-font-rem` / `fluid-space-rem` は `foundation/_functions.scss` を使用すること。
9. **レスポンシブ**: モバイルファーストで実装すること。SP 向けに整理したスタイルをベースとし、PC 用は `@include tb` 等のメディアクエリで差分を追加すること。PC 用スタイルを先に書き、SP で打ち消す書き方は避けること。
```
