# サーバー運用 改善提案

## 📌 今どんな問題が起きているか

サーバーの中に**テスト環境（制作中のサイトの確認用スペース）が増えすぎていて、管理が難しくなっています。**

具体的には…

- どの環境が何のために使われているか分からない
- 消してよいのか判断できず、誰かに確認が必要になる
- 不要なデータがどんどん溜まってサーバーの容量を圧迫している

サイトで使う画像やバックアップデータも時間とともに増え続けるため、  
**このまま放置すると、サーバーの容量がいっぱいになってサイトが正常に動かなくなるリスクがあります。**

---

## 💡 提案：サーバーを「用途別」に分ける

現在は1つのサーバーに何でも混在していますが、  
**役割を2つに分けて管理する**ことを提案します。


| サーバー       | 役割         | 内容                       |
| ---------- | ---------- | ------------------------ |
| **開発サーバー** | 制作中の案件専用   | 新しいサイトの確認・テスト用。リリース後は削除。 |
| **保守サーバー** | 保守契約中の案件専用 | リリース後も修正・確認が必要な案件のみ管理。   |


---

## 🔄 運用の流れ（イメージ）

```
開発サーバーでテスト環境を作成
        ↓
   本番サイトにリリース
        ↓
  ┌─────────────────┐
  │ 保守契約あり     │ → 保守サーバーへ移行して管理を継続
  │ 保守契約なし     │ → リリースから約30日後に削除
  └─────────────────┘
```

---

## 📋 運用ルール（案）

- テスト環境は **リリースから30日後に削除**する
- 「念のため残しておく」は原則なし
- 削除作業は**担当エンジニアがスケジュールに組み込んで**実施する

---

## ✅ この運用にするとどう変わる？

### 1. サーバーの中身が分かりやすくなる

「どこに何があるか」が用途ごとに整理されるため、  
エンジニア以外でも状況を把握しやすくなります。

### 2. 容量不足のリスクを防げる

不要な環境が自動的に整理されるため、  
突然サーバーがいっぱいになってサイトが止まる、といった事態を防げます。

### 3. 「消して大丈夫？」の確認が不要になる

ルールが明確になるので、毎回複数人に確認しなくても  
担当者が自分で判断して進められるようになります。

---

## 📝 補足

テスト環境はあくまで**一時的な作業スペース**です。  
長期間保存しておく必要は基本的にありません。

もし後から再確認が必要になった場合でも、  
ソースコード管理（Git）や本番環境のデータがあれば**再構築は可能**です。

必要に応じて調整はできればと思うので、
一度こういう形の運用も検討できればと思っています。
